# GitHub README 质量标准

> 用户对 README 质量要求很高。README 必须像"说明书"一样完整。

## 参考项目

| 项目 | README 大小 | Stars | 特点 |
|------|------------|-------|------|
| headroom | 20KB | 2k+ | 信息密度高，数据说话 |
| claude-obsidian | 37KB | 6.6k+ | 完整的技能文档 |
| swarmvault | 48KB | 556 | 详细的安装指南 |

## 必须包含的章节

1. **ASCII art logo** — 专业感
2. **徽章** — License、Python、框架版本
3. **快速导航** — 8+ 个链接
4. **What it does** — 8+ 个核心功能
5. **How it works** — 架构图
6. **Get started** — 30秒快速开始
7. **Proof** — 数据证明（具体统计数字）
8. **Compared to** — 竞品对比表格
9. **Scripts** — 脚本工具列表
10. **Requirements** — 环境要求表格（最低/推荐配置，分平台安装指南）
11. **FAQ** — 8+ 个常见问题
12. **When to use · When to skip** — 使用场景
13. **Troubleshooting** — 5+ 个常见问题解决方案
14. **Roadmap** — 路线图
15. **Contributing** — 贡献指南
16. **Related Projects** — 相关项目
17. **Acknowledgments** — 致谢
18. **License** — 许可证

## 关键原则

- **信息密度 > 字数** — 数据说话，不要写废话
- **用 `<details>` 折叠** — 详细内容折叠，保持页面简洁
- **竞品对比表格** — 清晰展示优势
- **Requirements 表格** — 最低/推荐配置，分平台安装指南
- **Troubleshooting** — 常见问题解决方案

## 工作流

1. 先在本地写好完整 README
2. 验证内容质量（大小、章节、格式）
3. 一次性上传到 GitHub
4. 上传后验证文件大小是否匹配

**不要**：先上传简化版 → 再更新完整版（浪费时间，容易出错）
